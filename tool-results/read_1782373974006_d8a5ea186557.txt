    85‚Üí## Incremental Static Regeneration
    86‚Üí
    87‚ÜíIncremental Static Regeneration (ISR) allows you to create or update content without redeploying your site. ISR has three main benefits for developers: better performance, improved security, and faster build times.
    88‚Üí
    89‚ÜíWhen self-hosting, (ISR) is limited to a single region workload. Statically generated pages are not distributed closer to visitors by default, without additional configuration or vendoring of a CDN. By default, self-hosted ISR does not persist generated pages to durable storage. Instead, these files are located in the Next.js cache (which expires).
    90‚Üí
    91‚ÜíTo enable ISR with Next.js in the app
    92‚Üí router, add an options object with a revalidate
    93‚Üí property to your fetch
    94‚Üí requests:
    95‚Üí
    96‚Üíapps/example/page.tsx
    97‚Üí
    98‚Üíexport default async function Page() {
    99‚Üí const res = await fetch('https://api.vercel.app/blog', {
   100‚Üí next: { revalidate: 10 }, // Seconds
   101‚Üí });
   102‚Üí 
   103‚Üí const data = await res.json();
   104‚Üí

   106‚Üí <main>

   166‚Üí## Server-Side Rendering (SSR)
   167‚Üí
   168‚ÜíServer-Side Rendering (SSR) allows you to render pages dynamically on the server. This is useful for pages where the rendered data needs to be unique on every request. For example, checking authentication or looking at the location of an incoming request.
   169‚Üí
   170‚ÜíOn Vercel, you can server-render Next.js applications through Vercel Functions.
   171‚Üí
   172‚ÜíTo summarize, SSR with Next.js on Vercel:
   173‚Üí
   174‚Üí- Scales to zero when not in use
   175‚Üí
   176‚Üí- Scales automatically with traffic increases
   177‚Üí
   178‚Üí- Has zero-configuration support for Cache-Control
   179‚Üí headers, including stale-while-revalidate
   180‚Üí
   181‚Üí- Framework-aware infrastructure enables automatic creation of Functions for SSR
   182‚Üí
   183‚ÜíLearn more about SSR

   185‚Üí## Streaming
   186‚Üí
   187‚ÜíVercel supports streaming in Next.js projects with any of the following:
   188‚Üí
   189‚Üí- Route Handlers
   190‚Üí
   191‚Üí- Vercel Functions
   192‚Üí
   193‚Üí- React Server Components
   194‚Üí
   195‚ÜíStreaming data allows you to fetch information in chunks rather than all at once, speeding up Function responses. You can use streams to improve your app's user experience and prevent your functions from failing when fetching large files.
   196‚Üí
   197‚Üí## Streaming with loading
   198‚Üí and Suspense
   199‚Üí
   200‚ÜíIn the Next.js App Router, you can use the loading
   201‚Üí file convention or a Suspense
   202‚Üí component to show an instant loading state from the server while the content of a route segment loads.
   203‚Üí
   204‚ÜíThe loading
   205‚Üí file provides a way to show a loading state for a whole route or route-segment, instead of just particular sections of a page. This file affects all its child elements, including layouts and pages. It continues to display its contents until the data fetching process in the route segment completes.
   206‚Üí
   207‚ÜíThe following example demonstrates a basic loading
   208‚Üí file:
   209‚Üí
   210‚Üíloading.jsx
   211‚Üí
   212‚Üíexport default function Loading() {
   213‚Üí return <p>Loading...</p>;
   214‚Üí}
   215‚Üí
   216‚Üíloading.tsx
   217‚Üí
   218‚Üíexport default function Loading() {
   219‚Üí return <p>Loading...</p>;
   220‚Üí}
   221‚Üí
   222‚ÜíLearn more about loading in the Next.js docs.
   223‚Üí
   224‚ÜíThe Suspense
   225‚Üí component, introduced in React 18, enables you to display a fallback until components nested within it have finished loading. Using Suspense
   226‚Üí is more granular than showing a loading state for an entire route, and is useful when only sections of your UI need a loading state.
   227‚Üí
   228‚ÜíYou can specify a component to show during the loading state with the fallback
   229‚Üí prop on the Suspense
   230‚Üí component as shown below:
   231‚Üí
   232‚Üíapp/dashboard/page.tsx
   233‚Üí
   234‚Üíimport { Suspense } from 'react';
   235‚Üíimport { PostFeed, Weather } from './components';
   236‚Üí 
   237‚Üíexport default function Posts() {
   238‚Üí return (
   239‚Üí <section>
   240‚Üí <Suspense fallback={<p>Loading feed...</p>}>
   241‚Üí <PostFeed />
   242‚Üí </Suspense>
   243‚Üí <Suspense fallback={<p>Loading weather...</p>}>
   244‚Üí <Weather />
   245‚Üí </Suspense>
   246‚Üí </section>
   247‚Üí );
   248‚Üí}
   249‚Üí
   250‚Üíapp/dashboard/page.jsx
   251‚Üí
   252‚Üíimport { Suspense } from 'react';
   253‚Üíimport { PostFeed, Weather } from './components';
   254‚Üí 
   255‚Üíexport default function Posts() {

   276‚Üí## Partial Prerendering
   277‚Üí
   278‚ÜíPartial Prerendering (PPR) pre-generates the static portions of a page and serves them from the cache, while it streams the dynamic portions in a single HTTP request. This allows you to serve content back to the user quickly while also allowing for user personalization without hurting performance.
   279‚Üí
   280‚ÜíAs of Next.js 16, PPR is no longer experimental. It is built into the Cache Components model, and you opt in by enabling cacheComponents
   281‚Üí in your next.config.ts
   282‚Üí:
   283‚Üí
   284‚Üínext.config.ts
   285‚Üí
   286‚Üíimport type { NextConfig } from 'next';
   287‚Üí 
   288‚Üíconst nextConfig: NextConfig = {
   289‚Üí cacheComponents: true,
   290‚Üí};
   291‚Üí 
   292‚Üíexport default nextConfig;
   293‚Üí
   294‚ÜíWith Cache Components, data is dynamic by default. You choose what to cache at the page, component, or function level with the use cache
   295‚Üí directive. Next.js then prerenders a static HTML shell and streams the dynamic content into it.
   296‚Üí
   297‚ÜíWhen a user visits a route:
   298‚Üí
   299‚Üí- A static route shell is served immediately, which makes the initial load fast.
   300‚Üí
   301‚Üí- The shell leaves holes where dynamic content streams in. The holes load in parallel, which allows dynamic or personalized content to fill in when it becomes available.
   302‚Üí
   303‚ÜíThis approach is useful for pages like dashboards, where unique, per-request data coexists with static elements such as sidebars or layouts. For example, this page caches its product list into the static shell with use cache
   304‚Üí, and streams the personalized greeting in at request time:
   305‚Üí
   306‚Üíapp/page.tsx
   307‚Üí
   308‚Üíimport { Suspense } from 'react';
   309‚Üíimport { cacheTag } from 'next/cache';
   310‚Üíimport { cookies } from 'next/headers';
   311‚Üí 
   312‚Üítype Product = { id: string; name: string };
   313‚Üí 
   314‚Üíasync function ProductsList() {
   315‚Üí 'use cache';
   316‚Üí cacheTag('products-list');
   317‚Üí 
   318‚Üí const products: Product[] = await fetch(
   319‚Üí 'https://api.example.com/products',
   320‚Üí ).then((res) => res.json());
   321‚Üí return (
   322‚Üí <ul>
   323‚Üí {products.map((product) => (
   324‚Üí <li key={product.id}>{product.name}</li>
   325‚Üí ))}
   326‚Üí </ul>
   327‚Üí );
   328‚Üí}
   329‚Üí 
   330‚Üíasync function Greeting() {
   331‚Üí const user = (await cookies()).get('user')?.value;
   332‚Üí return <p>Welcome back, {user}</p>;
   333‚Üí}
   334‚Üí 
   335‚Üíexport default function Page() {
   336‚Üí return (
   337‚Üí <section>
   338‚Üí <h1>Dashboard</h1>
   339‚Üí 
   340‚Üí {/* Cached: prerendered into the static shell */}
   341‚Üí <ProductsList />
   342‚Üí 
   343‚Üí {/* Dynamic: streamed in at request time */}
   344‚Üí <Suspense fallback={<p>Loading‚Ä¶</p>}>
   345‚Üí <Greeting />
   346‚Üí </Suspense>
   347‚Üí </section>
   348‚Üí );
   349‚Üí}
   350‚Üí
   351‚ÜíWhen you deploy to Vercel, the static shell is served with Incremental Static Regeneration (ISR), while Vercel Functions renders the dynamic holes and streams the content back in the same response.
   352‚Üí
   353‚ÜíIf you use PPR with Next.js 15, stay on your current canary and follow the
   354‚ÜíCache Components migration
   355‚Üíguide when
   356‚Üíyou move to Next.js 16, where PPR behaves differently.
   357‚Üí
   358‚ÜíSee the Cache Components docs to learn more.

   548‚Üí## Open Graph Images
   549‚Üí

   688‚Üí## Middleware
   689‚Üí
   690‚ÜíMiddleware is code that executes before a request is processed. Because Middleware runs before the cache, it's an effective way of providing personalization to statically generated content.
   691‚Üí
   692‚ÜíWhen deploying middleware with Next.js on Vercel, you get access to built-in helpers that expose each request's geolocation information. You also get access to the NextRequest
   693‚Üí and NextResponse
   694‚Üí objects, which enable rewrites, continuing the middleware chain, and more.
   695‚Üí
   696‚ÜíSee the Middleware API docs for more information.
   697‚Üí
   698‚ÜíTo summarize, Middleware with Next.js on Vercel:
   699‚Üí
   700‚Üí- Runs using Middleware which are deployed globally
   701‚Üí
   702‚Üí- Replaces needing additional services for customizable routing rules
   703‚Üí
   704‚Üí- Helps you achieve the best performance for serving content globally
   705‚Üí
   706‚ÜíLearn more about Middleware

   708‚Üí## Draft Mode
   709‚Üí
   710‚ÜíDraft Mode enables you to view draft content from your Headless CMS immediately, while still statically generating pages in production.
   711‚Üí
   712‚ÜíSee our Draft Mode docs to learn how to use it with Next.js.
   713‚Üí
   714‚Üí## Self-hosting Draft Mode
   715‚Üí
   716‚ÜíWhen self-hosting, every request using Draft Mode hits the Next.js server, potentially incurring extra load or cost. Further, by spoofing the cookie, malicious users could attempt to gain access to your underlying Next.js server.
   717‚Üí
   718‚Üí## Draft Mode security
   719‚Üí
   720‚ÜíDeployments on Vercel automatically secure Draft Mode behind the same authentication used for Preview Comments. In order to enable or disable Draft Mode, the viewer must be logged in as a member of the Team. Once enabled, Vercel's CDN will bypass the ISR cache automatically and invoke the underlying Vercel Function.
   721‚Üí
   722‚Üí## Enabling Draft Mode in Preview Deployments
   723‚Üí
   724‚ÜíYou and your team members can toggle Draft Mode in the Vercel Toolbar in production, localhost, and Preview Deployments. When you do so, the toolbar will become purple to indicate Draft Mode is active.
   725‚Üí
   726‚ÜíThe Vercel toolbar when Draft Mode is enabled.
   727‚ÜíUsers outside your Vercel team cannot toggle Draft Mode.
   728‚Üí
   729‚ÜíTo summarize, the benefits of using Draft Mode with Next.js on Vercel include:
   730‚Üí
   731‚Üí- Easily server-render previews of static pages
   732‚Üí
   733‚Üí- Adds additional security measures to prevent malicious usage
   734‚Üí
   735‚Üí- Integrates with any headless provider of your choice
   736‚Üí
   737‚Üí- You can enable and disable Draft Mode in the comments toolbar on Preview Deployments
   738‚Üí
